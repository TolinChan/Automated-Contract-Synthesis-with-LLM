# 调研报告（草稿）：基于 Move Prover 的 LLM 合约合成与修复闭环

## 1. 课题背景与动机

智能合约需要**高开发效率**（LLM 辅助编码）与**高保证**（数学级正确性）的平衡。生成式文件系统等工作（FAST ’26，SYSSPEC）表明可用**规约 + LLM** 生成实现，但其中 **LLM 充当验证器（SpecEval）** 存在幻觉风险，无法替代可靠后端证明器。过程反馈式修复（CodeNet4Repair）说明**编译器/验证器反馈**对修复质量至关重要，但**原始 SMT/求解器输出**未必适合直接输入 LLM。闭环验证生成（Clover，Dafny）与面向软件工程的 Agent 接口（SWE-agent）、言语强化（Reflexion）为**工具在环、多轮迭代**提供了可借鉴范式。

**本课题聚焦**：在 **Move + Move Prover** 场景下，构建 **Verifier-in-the-Loop**：规约/代码 → Prover →（经清洗的）反馈 → LLM 修复 → 再证明。

## 2. 核心技术路径

```
规约(Spec) / 意图 ──► LLM 生成或修改 Move 代码 ──► aptos move prove
        ▲                                              │
        │         反思 / Translator / 短反馈（ACI）     ▼
        └────────────── 解析日志与反例 ◄──────── 成功/失败
```

与 SWE-agent 的 ACI 类比：将 `prove` 封装为稳定工具，并对 stdout/stderr **结构化或截断**，避免噪声淹没上下文。与 Reflexion 类比：可将失败轮次摘要为**自然语言反思**写入记忆，减少重复错误。
