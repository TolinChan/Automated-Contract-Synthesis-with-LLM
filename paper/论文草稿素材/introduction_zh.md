## 1. Introduction

近年来，大型语言模型（LLM）驱动的代码生成能力取得了显著进展。从 GitHub Copilot [1] 到 Devin [2]、SWE-agent [3]，再到 Claude Code 和 Cursor，LLM coding agent 已经从实验室概念演变为工业界产品。在通用软件工程任务上，SWE-bench [4] 基准的解决率从 2023 年的 12% 快速提升至 2024 年的 43%，似乎预示着自动编程的临近。然而，这一数字背后存在可靠性隐患：Wang 等人 [5] 的 PatchDiff 分析发现，29.6% 的"已解决"问题存在行为差异，7.8% 的补丁虽然通过测试，但实际行为与 ground truth 不同。在涉及形式化验证的代码生成场景中，这一问题尤为突出——验证器的反馈是机器级别的（SMT 超时、验证条件失败、反例路径），远非单元测试"通过/不通过"那般二元明确。

当代码生成应用于区块链智能合约时，可靠性不足的后果尤为严重。去中心化金融（DeFi）协议的锁仓价值在 2025 年仍超过 500 亿美元，合约漏洞直接等同于可窃取的真实资金。Solidity 生态的历史已经多次验证这一点：The DAO 事件损失 6000 万美元（2016 年）[6]，Poly Network 遭到 6.11 亿美元攻击（2021 年）[7]，Nomad 桥损失 1.9 亿美元（2022 年）[8]。尽管行业已建立人工审计与自动化工具（Slither [9]、Mythril [10]、Certora [11]）相结合的防御体系，但审计周期长（2–4 周）、成本高（5–20 万美元），且仍无法覆盖所有执行路径。

我们前期的工作 SmartSpec [12] 尝试通过多模态规范（结合关系推理规则与时序逻辑）和反例引导的归纳合成（CEGIS）来自动生成 Solidity 合约，在 27 个合约上达到了 85% 的匹配率。然而，这一方法面临三个根本性局限：第一，Solidity 语言的动态派发和缺乏资源语义使得形式化规范难以表达，SmartSpec 只能支持一个高度受限的语言子集；第二，多模态规范本身要求开发者撰写复杂的推理规则，规范负担并未实质性降低；第三，CEGIS 循环产生的反例反馈需要人工解读和转化为修复提示，无法实现完全自动化。这些经验表明，在验证不友好的语言上强行做程序合成，会同时受限于语言语义和工具链的双重瓶颈。

Move 语言的设计为验证友好的程序合成提供了更合适的土壤。与 Solidity 相比，Move 通过线性类型系统实现了资源保留和所有权追踪的语义级保证，静态派发确保了编译时可判定性，强制字节码验证在部署前排除整类错误。Bartoletti 等人 [13] 的实证研究明确指出："Move is better suited for verification than Solidity"——在 Move 中"资源不丢失、所有权正确转移"这类安全属性在语言语义中天然成立，而在 Solidity 中"几乎不可验证，甚至不可表达"。这一语言层面的优势意味着，如果开发者只需提供形式化规范，程序合成系统生成满足规范的 Move 代码在理论上是可行的。

然而，Move 生态的实际情况表明，形式化验证工具链的使用门槛仍是核心障碍。Song 等人 [14] 的 MoveScan 工作对 Aptos 和 Sui 链上 37,302 个合约进行静态分析，发现了 97,169 个缺陷，其中算术溢出占 61.3%，跨模块状态污染在生产环境中高达 18.5%。然而，Move Prover 在这些真实缺陷中仅检出了 6.02%（在其支持的缺陷类型上 recall 为 83.12%）。这一落差并非因为验证技术本身不够强大，而是因为 Move Prover 需要开发者手写形式化规范（specification），而撰写规范往往比编写功能代码更为困难。即使有验证友好的语言设计，"把开发者意图转化为可被验证器理解的规范"这一过程仍然是人工密集且容易出错的。

这一瓶颈在真实世界的安全事件中已有体现。2024 年 11 月，Aptos 生态 DeFi 协议 Thala Labs 的 farming contract 因一个仅 2 行的补丁缺少关键状态检查而遭受 2550 万美元攻击 [15]——即使是有经验的开发者，在看似简单的修改中也会引入关键漏洞。形式化验证本可以阻止这类错误，但给定规范自动合成满足它的代码同样困难。以 Aptos 核心模块 `stake::update_performance_statistics` 为例，其规范要求维护模块级 ghost 变量、保持 while 循环不变式并处理 u64 算术溢出。即使向 LLM 提供完整的函数签名和规范（zero-shot），生成的代码仍因三个 Move-Prover 特有习惯用法而验证失败：ghost 变量必须通过 `spec` 代码块而非普通绑定更新；循环不变式必须嵌入 while 头部表达式；每次 `u64 += 1` 前需显式假设不溢出。实验表明，通用的错误反馈在 2 轮后仍导致超时，而一次性提供结构化诊断（人工列出全部 3 个问题）可在 1 轮内修复并通过验证（76.95 秒）。这说明，LLM 的代码生成能力并非瓶颈，瓶颈在于如何将验证器的机器级反馈转化为模型可执行的修复指令。

上述观察促使我们重新审视 LLM + 形式化验证的合成架构。首先，单次生成不可靠——已有研究表明即使最好的 agent 在超过 20% 的任务上持续失败 [16]，而验证器作为真值源能够捕捉语义违规（代码是否满足规范？），而非仅语法正确性。其次，让同一个模型同时生成代码和诊断自身错误会导致"为自己的错误辩护"——模型倾向于维护已生成的代码结构而非客观分析问题。我们在 `stake_update_perf` 上对比了多个模型的表现：Kimi 在 zero-shot 和反馈循环中均失败；Claude Opus 4.7 在 1 轮结构化诊断后即通过验证（72.7 秒）；GPT 5.5 虽然在 zero-shot 下看似通过，实则遗漏了关键逻辑分支 `failed_proposer_indices`，只是因规范未完全约束而侥幸通过。这揭示了 Pass@1 指标的致命缺陷：验证器通过不等于代码语义完整。因此，合成系统的核心挑战不在于选择更强的代码生成模型，而在于构建高质量的结构化诊断机制。

基于以上分析，本文提出 MoVES（Move Verified Contract Synthesis），一个面向 Move 语言的、基于形式化验证反馈循环的智能合约合成系统。本文的核心贡献包括：

（1）**诊断分类学与双角色架构。** 通过对 Move Prover 验证失败的系统分析，提出面向 Move 规范错误的诊断分类学（diagnosis taxonomy），并设计 codegen-diagnoser 双角色架构，由独立模型负责代码生成与错误诊断，避免单一模型的自我辩护偏差。

（2）**Verifier-in-the-Loop 合成机制。** 实现基于验证器反馈的闭环合成流程，验证结构化诊断反馈相比通用反馈在修复效率上的显著提升，并证明该架构在 1 轮反馈内即可达到接近人工诊断的修复效果。

（3）**跨模型能力分析与指标反思。** 通过对比 Kimi、Claude Opus 4.7 和 GPT 5.5 在相同任务上的表现，揭示当前 LLM 在形式化验证代码合成中的关键瓶颈在于诊断质量而非生成能力，并发现 Pass@1 指标在验证场景下的系统性缺陷。

（4）**实现与评估。** 在 Aptos 标准库和真实世界合约上实现 MoVES 原型系统，系统评估其在多种 Move-Prover 习惯用法上的合成效果，证明结构化诊断反馈可显著提升验证通过率。
