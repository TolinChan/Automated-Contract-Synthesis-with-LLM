# MoVES：基于规范驱动的智能合约合成与验证器闭环反馈

> 完整论文草稿 —— ASE 投稿格式
> 目标：Automated Software Engineering (ASE)，完整研究轨道

---

## 摘要

生成满足形式化规范的程序实现是软件工程的长期目标。大语言模型（LLM）可以根据自然语言或形式化规范生成代码，但生成的代码经常无法通过形式化验证——特别是对于具有原生验证支持的语言，如 Aptos Move。我们观察到，瓶颈通常不在于 LLM 的*代码生成*能力，而在于它无法解释低级别的验证器输出并将其转化为可操作的、领域特定的修复方案。

我们提出 **MoVES**，一个面向 Move 智能合约的验证器闭环合成流水线。给定函数签名和固定的 Move-Spec 形式化规范，MoVES 反复合成函数体并通过 Move Prover（Boogie/Z3）进行验证。当验证失败时，一个专用的**诊断器（diagnoser）**角色将失败分类为 Move-Prover 特定惯用法（idiom）的分类体系，并开出结构化的修复指令，反馈给独立的**代码生成（code-generation）**角色进行下一轮迭代。这种双角色分离防止了生成器为自己的错误辩护，并保持反馈有界且可操作。

对五个 Aptos 框架函数的最小可行性测试表明，零样本（zero-shot）LLM 合成通过了 4/5 的用例；唯一失败的用例——一个需要幽灵变量更新、while 头循环不变式和溢出假设的复杂函数——无法通过通用错误反馈修复，但当诊断器显式命名所需的三个惯用法时，一轮即可修复。这些结果将*代码生成能力*（足够）与*诊断质量*（瓶颈）分离开来，并推动了 MoVES 核心的结构化、领域特定反馈循环。

**关键词：** 智能合约合成、形式化验证、Move Prover、LLM 智能体、反馈循环

## 1. 引言

智能合约管理链上的高价值资产；一个单一的错误可能导致不可逆的财务损失。形式化验证提供了正确性的数学保证，但编写实现和形式化规范都需要劳动密集型的专业知识。从形式化规范自动合成已验证的实现将显著降低这一门槛。

大语言模型在从自然语言描述生成代码方面已显示出前景 [1,2]，最近的工作已将其扩展到规范驱动的生成 [3,4]。然而，当目标语言包含内置的形式化验证器时——如 Aptos Move 的 Move Prover [5]——一个"看起来正确"的生成体仍可能因微妙的领域特定要求而失败验证（例如，幽灵变量簿记、循环不变式放置、算术溢出假设），这些要求 LLM 在其训练数据中未曾遇到。

一种自然的回应是闭环：运行验证器，将其错误输出反馈给 LLM，并要求修订体。先前工作已在其他领域探索了这一模式 [6,7,8]，但两个问题仍未得到充分解决：

1. **原始验证器输出级别太低。** Move Prover 发出 Boogie/Z3 级别的消息（验证条件失败、SMT 超时、反例跟踪）。将这些直接作为"反馈"输入 LLM 会压垮上下文窗口，且不会揭示所需的高级代码转换。
2. **要求一个 LLM 同时生成和自我批评混淆了两个不同的任务。** 被要求修复自己输出的生成器倾向于进行局部的、语法层面的修补，而非识别缺失的领域惯用法；失败模式和修复往往是正交的技能。

我们通过**双角色分离**和**结构化领域特定诊断**来解决这两个问题。MoVES 将工作拆分为*代码生成*角色（生成体）和*诊断器*角色（分析验证器输出并用 Move-Prover 惯用法的词汇开出修复方案）。诊断器的输出不是原始 Prover 的标准错误输出；它是一个类型化记录 `{CATEGORY, ROOT_CAUSE, FIX_INSTRUCTION}`，用领域术语表达，代码生成角色可以据此行动。

### 1.1 动机示例

考虑来自 Aptos `stake` 模块的 `update_performance_statistics`，一个更新验证者性能计数器的函数。其规范（在我们的实验中始终保持固定）声明了两个模块级幽灵变量、一个对失败提议者索引的 while 循环，以及将更新状态与输入快照关联的后置条件。

当我们要求 LLM（`kimi-for-coding`）仅从签名和规范生成函数体（零样本）时，它生成了一个编译通过但在 72 秒后超时而验证失败的体。生成的体缺少三个 Move-Prover 特定的惯用法：

- **惯用法 1 —— 幽灵变量更新。** 规范引用 `global ghost_valid_perf` 和 `global ghost_proposer_idx`；体必须用 `spec { update ghost_valid_perf = validator_perf; }` 块初始化它们，而不是常规的 `let` 绑定。
- **惯用法 2 —— while 头循环不变式放置。** 循环不变式必须放置在 while 头表达式内部（`while ({ spec { invariant ... }; cond })`），而不是循环体内。LLM 将它们放在体内，触发了编译级 Prover 错误。
- **惯用法 3 —— 溢出假设。** 每个 `u64 += 1` 必须前面加上 `spec { assume X + 1 <= MAX_U64; };`，否则 Prover 生成无界溢出验证条件并超时。

然后我们运行通用反馈循环（B6）：Prover 错误被反馈给 LLM，提示要求它诊断并修复。一轮反馈后体发生了变化，但三个惯用法中的两个仍然错误；错误类型从超时变为"循环不变式必须在循环头开头声明"。第二轮后结果仍然是失败。循环确实在取得*一些*进展，但通用诊断器不会揭示惯用法级别的修复，因为它没有 Move-Prover 检查清单。

最后，我们手写一份结构化诊断，显式命名所有三个惯用法并附带代码级指令，通过*相同*的代码生成提示反馈，结果体在第一次尝试时以 76.95 秒通过验证。生成的体几乎与参考实现逐行匹配。

**洞察。** LLM 的*代码生成*能力足以为此复杂函数生成验证体。瓶颈在于*诊断*步骤：没有将原始 Prover 输出桥接到可操作代码转换的领域特定分类体系，反馈循环无法收敛。

### 1.2 贡献

我们做出以下贡献：

1. **验证器闭环合成的双角色架构。** 我们将代码生成与错误诊断分离为具有不同契约的不同 LLM 角色，防止生成器为自己的错误辩护，并使瓶颈（诊断质量）独立可测量。
2. **面向 Move Prover 的领域特定诊断分类体系。** 我们识别了一组有限的失败模式和规范修复（幽灵变量更新、while 头循环不变式放置、溢出假设），将原始验证器输出桥接到可操作的代码生成指令。
3. **经验的关注点分离。** 对五个 Aptos 框架函数的最小可行性测试表明，零样本代码生成在 4/5 上成功；唯一失败通过手动结构化诊断在一轮内修复，但通用反馈无法修复。这证明代码生成能力不是限制因素——诊断质量才是。

## 2. 背景

### 2.1 Move 语言与 Move Prover

Move 是一种为 Aptos 区块链开发的面向资源的智能合约语言 [9]。资源是线性类型，不能隐式复制或丢弃，使资产跟踪适合静态分析。Move Prover [5] 是集成到 Aptos 工具链中的形式化验证器；它将 Move 代码与 Move-Spec（MSL）注解一起翻译成 Boogie 中间验证语言，然后由 Z3 SMT 求解器处理。

Move 函数可以携带 `spec fun` 块，声明前置条件（`requires`）、后置条件（`ensures`）、中止条件（`aborts_if`）、框架条件（`modifies`）和循环不变式（`invariant`）。Prover 从函数体和规范块生成验证条件（VC）；如果 Z3 证明所有 VC，则函数被验证。否则，Prover 发出错误跟踪或超时。

### 2.2 规范驱动合成任务

我们采用以下任务定义，将范围缩小到足以获得清晰的正确性信号：

> **输入。** 函数签名、固定的 `spec fun` 块和模块上下文（导入、结构体、兄弟函数签名、模块级幽灵变量声明）。
> **输出。** 一个函数体，当拼接到签名中时，根据 `aptos move prove` 满足规范。
> **不变量。** 规范块是只读的；唯一可编辑的表面是函数体。

这个定义是故意的：它将规范合成从循环中移除，给了我们一个确定性裁判（Prover 退出码）和一个有界的编辑表面（一个函数体）。扩展到规范合成是未来工作（第 7 节）。

## 3. 相关工作

我们沿两个维度分类相关工作：**规范驱动 vs. 错误驱动**输入，以及**规范合成 vs. 代码合成**任务。

### 3.1 规范驱动合成

**MSG**（Zhang 等，ASE'25）[10] 使用模块化多智能体架构（`aborts_if`、`ensures`、`modifies` 和循环不变式的独立智能体）从代码生成 Move 规范。它是我们目标语言最接近的先前工作，但解决了逆问题：代码 → 规范，而非规范 → 代码。其 Prover 反馈循环精炼生成的规范，而非实现。

**PropertyGPT**（Chu 等）[11] 使用检索增强生成来为 Solidity 生成 Certora/Echidna 属性。其循环架构（LLM → 验证器 → 错误反馈 → LLM 修订）在结构上与我们的验证器闭环设计相似，但 LLM 的任务是发明属性，而非针对现有规范生成实现。

**Clover**（LaChance 等）[12] 一起生成代码和注解，强制执行代码、注解和文档字符串之间的一致性。它依赖 GPT-4 作为文档字符串边缘的一致性检查器，创造了我们方法避免的"LLM 评判 LLM"漏洞。

**SysSpec**（Liu 等，FAST'26）[3] 展示了规范驱动的文件系统生成，验证了形式化规范可以在系统领域指导 LLM 输出。它启发了我们的规范驱动输入假设，尽管其目标（文件系统）不涉及外部验证器裁判。

### 3.2 错误驱动修复

**RePair**（Le 等）[6] 给 LLM 提供错误代码加编译器/测试错误日志并要求修复。这与我们的方法根本不同：我们从结构化规范开始并生成实现，而非修补现有错误程序。

**Reflexion**（Shinn 等）[7] 和 **SWE-agent**（Yang 等）[8] 探索了带有工具使用和语言强化的多轮智能体循环。这些启发了我们的循环控制设计（轮次预算、持久化、指标分离），但两者都没有解决将形式化验证器输出转化为可操作代码修复的具体挑战。

### 3.3 定位

| 维度 | MSG | PropertyGPT | RePair | **MoVES** |
|---|---|---|---|---|
| 任务 | 代码 → 规范 | 属性合成 | 错误修复 | **规范 → 代码** |
| 输入 | 代码 + 静态分析 | 检索的属性 | 错误代码 + 错误 | **形式化规范** |
| 裁判 | Move Prover | Certora/Halmos/Echidna | 测试/编译器 | **Move Prover** |
| 反馈 | 通用 Prover 错误 | 属性精炼 | 回滚到最佳步骤 | **结构化惯用法级诊断** |
| 角色 | 多智能体（规范子句） | 单 LLM + RAG | 执行器 + 评判器 | **代码生成 + 诊断（分离）** |

MoVES 的独特之处在于（1）规范到代码的方向，（2）具有领域特定诊断器的双角色分离，以及（3）没有 LLM 作为评判器的硬验证器裁判。

## 4. 设计

### 4.1 架构概述

MoVES 是一个具有两个 LLM 角色和一个确定性裁判的反馈循环：

```
冻结输入
  ├─ 函数签名
  ├─ 形式化规范块（只读）
  └─ 模块上下文（导入、结构体、兄弟签名、幽灵声明）
         │
         ▼
   ┌──────────────┐     body_k
   │  代码生成    │ ───────────────┐
   │  LLM         │                │
   └──────────────┘                ▼
         ▲              ┌───────────────────┐
         │              │  拼接 + 验证器    │
    反馈提示            │  （确定性        │
         │              │   裁判）          │
         │              └───────────────────┘
         │                       │
         │            通过 / 失败+输出
         │                       │
         │              ┌────────┴─────────┐
         │              │                  │
         │           停止              ┌──────────┐
         │           （成功）          │ 诊断     │
         │                             │ LLM      │
         │                             │(分类 + 处方)│
         │                             └──────────┘
         │                                  │
         └──────────────────────────────────┘
                    diagnosis_k
```

**代码生成角色。** 接受冻结输入加上可选的先前体和可选的结构化诊断。输出用可识别标记围栏的函数体。其契约是：*生成满足规范的 Move 代码，当拼接到函数中时*。它不被要求分析先前尝试为何失败。

**验证器步骤。** 将生成的体拼接到规范源代码树的干净副本中并运行 `aptos move prove`。验证器是**通过/失败的唯一真实来源**；没有 LLM 决定正确性。这消除了幻觉成功。

**诊断角色。** 接受规范、签名、模块上下文、失败体和 Prover 标准输出/标准错误。输出类型化记录 `{CATEGORY, ROOT_CAUSE, FIX_INSTRUCTION}`，其中修复指令用**领域惯用法**（而非原始 Prover 术语）表达。其契约是：*将验证器输出转化为失败模式的分类体系并开出代码级修复*。它不重写体。

### 4.2 设计原则

**P1 —— 两个 LLM 角色，每个单一职责。** 一个 LLM 生成；另一个分析。从不在同一个调用中。被要求在同一轮中自我纠正的生成器倾向于为其先前输出辩护。分离角色使瓶颈独立可测量：如果当诊断是手写时代码生成成功，则诊断器是瓶颈。

**P2 —— 验证器是唯一真实来源。** 循环中的 LLM 从不决定通过/失败。验证器退出码是契约。

**P3 —— 规范是不变量；体是唯一编辑表面。** 循环的自由度是一个函数体。这使拼接机械化、工作区重置廉价，并防止"通过削弱规范获得成功"。

**P4 —— 领域桥梁，而非通用错误散文。** 诊断器分类为有限分类体系并用领域惯用法开出修复。没有这一点，诊断器就变成了冗长的错误重新格式化器，代码生成角色得不到可操作的指导。

**P5 —— 轮次级工件持久化。** 每一轮都写入其提示、模型响应、提取的体、验证结果和诊断。循环在事后完全可重放。

### 4.3 循环控制

| 关注点 | 选择 |
|---|---|
| 预算 | `feedback_rounds`（我们的实验中为 1 或 3）加上 1 轮 round-0 尝试 |
| 成功停止 | 验证器退出码为 0 的第一轮 |
| 失败停止 | 预算耗尽 |
| 指标 | `Pass@1`（仅 round 0）和 `rounds_to_success`（从不混合） |

## 5. 评估

### 5.1 研究问题

- **RQ1：** 零样本 LLM 合成在规范驱动的 Move 体生成上表现如何？
- **RQ2：** 通用验证器反馈能提高成功率吗？
- **RQ3：** 结构化领域特定诊断能否修复通用反馈无法修复的用例？

### 5.2 实验设置

| 项目 | 值 |
|---|---|
| 模型 | `kimi-for-coding`（通过 OpenAI 兼容 API） |
| 验证器 | `aptos move prove`, Aptos CLI 9.1.0, Boogie 3.5.1, Z3 4.13.0 |
| 每 VC 超时 | 40 秒 |
| 工作区 | 每次验证运行前从规范源完整重置 |

### 5.3 数据集

来自 `aptos-framework` 的五个函数，选择以跨越规范复杂度：

| 函数 | 模块 | 规范复杂度 |
|---|---|---|
| `chain_id_get` | `chain_id.move` | 简单（1 行） |
| `chain_id_initialize` | `chain_id.move` | 简单（`signer`, `aborts_if`） |
| `coin_extract` | `coin.move` | 中等（结构体变更 + `aborts_if`） |
| `block_initialize` | `block.move` | 中高（~25 行） |
| `update_performance_statistics` | `stake.move` | 复杂（幽灵变量 + while/不变式 + 溢出，~50 行） |

### 5.4 基线

- **B1（零样本）。** 提示 = 签名 + 规范。无上下文。
- **B3（带模块上下文的零样本）。** 提示 = 签名 + 规范 + 导入、结构体、兄弟签名。
- **B6（自动反馈，1 轮）。** Round 0 = B3。如果失败，通用诊断提示 → 反馈 → round 1。
- **B7（自动反馈，3 轮）。** 最多 3 轮通用诊断反馈。
- **手动诊断（结构化诊断，1 轮）。** 手写诊断命名 Move-Prover 惯用法，通过相同的代码生成提示反馈。

### 5.5 结果

**RQ1 —— 零样本性能。**

| 函数 | B1 | B3 |
|---|---|---|
| `chain_id_get` | 通过 (15.6 秒) | 通过 (16.4 秒) |
| `chain_id_initialize` | 通过 (16.0 秒) | 通过 (16.8 秒) |
| `coin_extract` | 通过 (27.6 秒) | 通过 (22.3 秒) |
| `block_initialize` | 通过 (42.2 秒) | 通过 (39.1 秒) |
| `update_performance_statistics` | 失败 (72.2 秒) | 失败 (71.2 秒) |

**通过率：4/5 (80%)** 在 B1 和 B3 中均如此。添加模块上下文不会提升失败用例。

**RQ2 —— 通用反馈。**

| 函数 | B6 `rounds_to_success` | B7 `rounds_to_success` |
|---|---|---|
| `chain_id_get` | 1 | 1 |
| `chain_id_initialize` | 1 | 2 |
| `coin_extract` | 1 | 1 |
| `block_initialize` | 1 | 1 |
| `update_performance_statistics` | 失败 | 失败 |

通过率保持 4/5。通用反馈无法修复唯一失败用例。自动诊断器误分类根本原因（例如，推荐规范修改，这是被禁止的）且不会揭示所需的三个惯用法。

**RQ3 —— 结构化诊断。**

对于 `update_performance_statistics`，手写结构化诊断显式命名了三个惯用法（第 1.1 节），通过相同的代码生成提示反馈。结果：

| 指标 | 值 |
|---|---|
| 通过 | **是** |
| 退出码 | 0 |
| 验证时间 | **76.95 秒** |
| 体质量 | 匹配参考实现；三个惯用法均存在 |

### 5.6 分析

结果分离了两个关注点：

1. **代码生成能力足够。** `kimi-for-coding` 在获得精确、命名惯用法的诊断时，可以为最难的函数生成验证体。
2. **瓶颈在于诊断步骤。** 通用 Prover 输出摘要提示不会揭示 Move-Prover 惯用法；领域特定诊断会。

这验证了双角色架构：当诊断器角色完成其工作时，代码生成角色有效。下一步是将诊断器自动化以匹配手动诊断结果，无需人工干预。

---

## 6. 讨论

### 6.1 有效性威胁

**外部有效性。** 数据集是来自一个框架的五个函数。虽然集合跨越了规范复杂度，但推广到更大的基准（例如，完整的 MSG 基准套件）是未来工作。

**结构有效性。** 我们使用 `aptos move prove` 退出码作为唯一正确性标准。这是可靠的（Prover 是可信工具），但可能错过规范未覆盖的运行时行为。我们的任务定义明确将范围限制为规范满足。

**内部有效性。** 手动诊断结果证明了架构的潜力，但没有证明*自动*诊断器有效。带有惯用法检查清单的自动诊断器尚未匹配手动结果（feas_run_03，未包含在此交付物中）。架构是可靠的；诊断器提示工程正在进行中。

### 6.2 开放问题

**诊断器自动化。** 惯用法库目前部分嵌入诊断提示中。扩展到完整基准需要（a）可检索的工作示例知识库，或（b）从过去失败跟踪中学习分类体系。

**单角色消融。** 受控的"一个 LLM 做两件事"基线将经验验证 P1（双角色分离），区分架构收益和提示质量。

**少样本提示。** 在 round-0 提示中添加 1-2 个正确的规范→代码示例能否在没有反馈循环的情况下解决惯用法问题？此基线已计划但尚未运行。

---

## 7. 结论

我们提出了 MoVES，一个面向 Move 智能合约的验证器闭环合成流水线。其核心设计——代码生成与诊断之间的双角色分离，以及将原始 Prover 输出桥接到可操作修复的领域特定惯用法分类体系——是由一个经验观察驱动的：LLM 可以为复杂的 Move 函数生成验证代码，但仅在反馈显式命名所需的领域惯用法时。通用反馈循环在同一用例上失败。

对五个 Aptos 框架函数的最小可行性测试支持这一声明：4/5 零样本通过；唯一失败通过结构化手动诊断在一轮内修复，但通用自动反馈无法修复。这将代码生成能力与诊断质量分离开来，并将诊断器确立为扩展到更大基准的关键组件。

**未来工作**包括（1）将诊断器自动化以匹配手动诊断质量，（2）扩展到 50+ 函数的基准，（3）运行单角色和少样本消融，以及（4）将架构移植到其他验证语言目标（例如，Rust + Kani，Dafny）。
