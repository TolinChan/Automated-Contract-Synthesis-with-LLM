# 文献笔记：RePair / CodeNet4Repair（2408.11296）

## 元信息

- **标题**：*RePair: Automated Program Repair with Process-based Feedback*
- **作者**：Yuze Zhao 等，中国科学技术大学等
- **版本**：arXiv:2408.11296v1（cs.SE），2024-08-21
- **代码与数据**：https://github.com/TnTWoW/RePair

## 一句话问题

在 **小于 100B 甚至约 15B** 规模的模型上，如何通过**过程监督**与**环境反馈**，让程序修复接近大闭源模型的效果？

## 核心贡献

1. **数据集 CodeNet4Repair**：从 CodeNet 清洗而来，包含**题目描述**、**时空限制**、**从错误到 AC 的多步提交轨迹**、执行状态（内存、CPU、代码规模等）；强调与 DeepFix、Review4Repair、Bug2Fix 等相比具备 **测试用例 + 描述 + 多步过程**（测试集约 10144 条复杂修复过程于测试集）。
2. **框架 RePair**：**SFT** 让模型理解修复任务 → **奖励模型（RM）** 模仿「编译器+测试」对程序状态打分 → **PPO** 强化学习，actor 逐步修复、critic 给反馈；推理时**迭代生成**直到奖励不再提升或达步数/耐心上限。
3. **实证**：在 CodeNet4Repair 上，15B 级 **StarCoderBase** 经本方法后在 pass@1/@3/@5 上**优于其他开源模型**，并与 **GPT-3.5、Claude2** 等商用模型**可比**。

## 过程反馈的定义（与竞赛行为对齐）

- **Figure 1**：选手根据题意与限制写初解 → 提交得 **TLE/MLE/WA** 等 → 多轮改进至 **AC**；强调**单步大改**往往编辑距离大、难度大，**多步小改**更符合人类行为。
- **与 outcome-based 对比**：传统 APR/LLM 多视为**单步** seq2seq；本文强调**每一步都需要监督信号**，且 APR 中间步往往是「错误监督」，不能直接当数学推理里那种「步步须正确」的过程标签用——因此采用 **RM + RL** 而非简单模仿每步 gold 编辑。

## 方法细节

### SFT（§3.1）

- 基座：**StarCoderBase 15.5B**，8K 上下文；提示模板见附录 Figure 7（问题描述 + 错误程序 → 修复程序；可选加入测试反馈的多轮模板）。

### 奖励模型（§3.2.1）

- **不用**训练时逐步跑真实编译器+测试（吞吐崩溃：tensor↔CPU 编解码与串行评测）；改用 **RM 作为 virtual tool**。
- **偏序**（质量从高到低）：**AC > PE > WA = TLE = MLE > CE > RE**。理由简述：优先修 RE（更偏逻辑/严重）等。
- **训练**：对同一题多个程序状态做 **pairwise ranking loss**（式 2）；RM 架构为较小自回归模型 + 投影层输出标量。
- **推理**：若无隐藏测试，纯 RM 可支持泛化；文中亦讨论与真实环境的关系。

### RL（§3.2.2–3.3）

- **PPO**：LM 为 actor，RM 为 critic；目标含奖励增量与 KL 约束（式 3）。
- **推理 Algorithm 1**：每步生成 `x_{t+1}`，算 `r_{t+1}`，若 **Δr ≤ 0** 则耐心计数 +1，否则清零；结束取 **回滚到未改善前的最优步**（`return x_{t-p}`）。

### 数据划分与规模（Table 2）

- Train：563 题 / 94062 条 repair 记录；Test：61 题 / 10144 条记录；等。

## 实验结论摘要

- **主表（Table 3）**：按题目通过率分 Easy/Medium/Hard；**RePair（15B）** 在各难度与 pass@k 上整体强于表格中开源基线，接近 PaLM/GPT-3.5/Claude2 等。
- **消融（Table 4）**：无过程监督或无反馈均明显变差；**pairwise** RM 优于 point-wise / list-wise。
- **§4.7**：同数据上仅 SFT 其他开源模型有涨，但 **+过程反馈** 额外放大收益。
- **§4.8**：小模型对 **显式 prompt 里的编译器/测试反馈** 仍难充分利用；**ChatGPT** 在显式反馈下多步更好——暗示**反馈形式与模型规模**共同决定闭环效果。

## 局限（§7）

- CodeNet4Repair 聚焦**竞赛式 Python** 修复，未覆盖多语言与全部软件工程场景；作者认为与工程场景仍有相似性，可作部分迁移评估。

## 与你课题（Move + Move Prover + Agent）的对接

- **直接借鉴叙事**：**编译器/验证器在环的多步修复**；**奖励或状态偏序**思想可类比「Prover 失败类型 / 反例阶段」的分级信号（需你自己定义与 Move 日志对齐的序或特征）。
- **关键难点（可写进开题）**：RePair 的反馈是 **WA/TLE** 等相对「类型化」信号；**Move Prover / Z3** 日志更长、更偏证明论与求解器内部——对应你计划中的 **错误翻译 / 结构化反馈**，小模型或固定 prompt 更难吃透（与 §4.8 呼应）。
- **训练 vs 推理**：全文以 **微调+RM+RL** 为主；若你做 **零微调 Agent**，仍可借用其 **「多步 + 真实工具反馈」** 的实验叙事与对比逻辑。

## 引用时可用的概括句

- RePair 用 **过程反馈 + 学习到的 critic** 缓解小模型**单步大补丁**能力不足，使开源中等规模模型在竞赛级修复上逼近商用 LLM。
- 数据集层面，CodeNet4Repair 填补了「**带测试与题面、多步轨迹**」的 APR 数据空白。
